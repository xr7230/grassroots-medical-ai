# 基层医疗AI文书智能体 — 落地路径

> 本文档是肖锐（2026）《基层医疗AI需求分析与设计转向》论文的工程落地延伸。
> 产品定位、目标用户、设计原则均与论文结论保持一致。

## 一、论文结论回顾

基于52条基层医务工作者真实评论的扎根理论编码：

| 需求层次 | 占比 | 核心诉求 |
|----------|------|----------|
| 效率减负 | **40.4%** | 文书工作自动化减负 |
| 基础设施 | 26.9% | 系统稳定、低门槛部署 |
| 话语权 | 21.2% | 参与AI治理 |
| 智能诊断 | 11.5% | 诊断辅助（非刚需） |

**核心发现**：现有市面AI产品存在"重诊断、轻文书"的结构性价值错配。

**设计转向主张**：从"诊断智能体" → "文书智能体"，以"安静后台助手"和"垂直场景极致单品"两种形态落地。

## 二、产品定位

### 2.1 目标用户

| 优先级 | 用户 | 依据 |
|--------|------|------|
| 🥇 | 乡镇卫生院 / 社区卫生中心 | 接诊压力大、文书负担重、信息化基础薄弱 |
| 🥈 | 村卫生室 / 个体诊所 | 单人执业，无专职文书，最需要自动化 |
| 🥉 | 县级医院门诊 | 日接诊量高，文书工作挤占诊疗时间 |

**明确不做**：三甲医院住院病历（已有成熟HIS/EMR系统，论文指出"HIS集成门槛"是基层部署的主要障碍）。

### 2.2 与论文竞品分析的差异化

论文评测的四款产品均存在基层适配性问题：

| 竞品 | 论文指出的问题 | 本产品的回应 |
|------|---------------|-------------|
| 讯飞智医助理 | 依赖HIS集成，部署门槛高 | 独立运行，零HIS依赖 |
| 百度灵医智惠 | 功能分散，双入口割裂 | 单一入口，专注文书 |
| 联影智能 | 适配高端设备，基层无CT/MRI | 仅需文本输入 |
| 左手医生 | C端症状自查，非文书工具 | B端文书生成 |

### 2.3 设计原则（与论文一致）

1. **安静后台** — 不嵌入HIS，不打断医生工作流
2. **低门槛** — 浏览器打开即用，仅需DeepSeek API Key
3. **给方向不给定论** — 诊断标注"可能/待排"，不做百分百确诊
4. **安全先行** — 17类输入守卫，隐私脱敏，不编造数据

## 三、落地路径

### 3.1 第一阶段：开源验证（当前）

**目标**：以开源项目形式发布，验证"文书智能体"设计转向的技术可行性。

- GitHub 公开仓库，MIT 协议
- 配套论文引用，形成"研究→设计→实现"闭环
- 面向基层医生的演示视频 + 使用文档

**关键指标**：
- 论文答辩时可现场演示
- 为后续研究提供参考实现
- 接受社区反馈和贡献

### 3.2 第二阶段：基层试点（论文后续）

**目标**：在真实基层场景中验证产品效果。

- 选取 3-5 家乡镇卫生院/社区卫生中心作为试点
- 收集使用数据：日均生成病历数、医生满意度、文书时间节省
- 对照论文需求编码，验证"效率减负40.4%"是否被满足

**关键指标**：
- 单份病历生成时间 < 30秒
- 医生满意度 > 4分（5分制）
- 文书时间节省 > 50%

### 3.3 第三阶段：推广应用

**路径A — 安静后台助手**（论文方案一）：
- 保持独立运行，不做HIS集成
- 浏览器书签即可使用
- 适合信息化基础薄弱的基层站点

**路径B — 垂直场景极致单品**（论文方案二）：
- 聚焦入院记录 / 出院小结 / 随访记录三类文书
- 不做诊断、不做用药、不做影像
- 与现有基层公卫系统（如基层HIS）通过文本导入/导出松耦合

## 四、推广策略

### 4.1 学术推广
- 论文发表后，产品作为方法验证开源
- 在医学信息学、基层卫生管理相关会议展示
- 与基层卫生研究机构合作开展效果评估

### 4.2 基层渠道
- 卫健委基层卫生处 / 县域医共体
- 基层医生继续教育平台
- 丁香园基层医生社区

### 4.3 政策对齐
- 国家卫健委"优质服务基层行"活动
- 电子病历分级评价（基层版）
- 家庭医生签约服务中的文书规范化需求

## 五、风险与限制

| 风险 | 应对 |
|------|------|
| LLM生成内容准确性问题 | 标注"仅供参考"，人工审核兜底 |
| 基层网络条件差 | 纯文本API调用，带宽需求极低（<10KB/次） |
| 医生使用习惯差异 | 支持文本/语音输入，零学习成本 |
| 论文结论基于有限样本 | 标注"52条样本，结论外推需验证" |

## 六、与论文的关系

```
论文：发现问题 → 需求编码 → 设计转向建议
  ↓
产品：实现"文书智能体"设计转向
  ↓
反馈：试点数据验证论文结论
```

本产品是论文从理论到实践的桥梁。论文提供了方向，产品提供了证明。
